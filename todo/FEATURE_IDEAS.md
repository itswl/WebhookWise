## 1. 告警智能降噪 + 根因分析

### 💡 核心创意

**问题**：现在只是去重，但没有识别告警之间的因果关系

**场景示例**：
```
数据库挂了 → 引发100个应用告警
当前行为：100条告警都通知
期望行为：识别为"根因：数据库故障" + 99个衍生告警
```

### 🎯 解决方案

**告警关系图构建**：
```
[数据库连接超时] ← 根因
  ├─ [服务A响应慢]
  ├─ [服务B报错]
  └─ [API 5xx激增]

AI分析提示：
"检测到告警风暴：根因可能是数据库故障，已抑制 99 个衍生告警"
```

### 🛠️ 实现思路

1. **时间窗口关联**
   - 在 5 分钟窗口内的告警做关联分析
   - 识别告警的先后顺序和依赖关系

2. **依赖图谱构建**
   ```python
   # 数据库新增表
   CREATE TABLE alert_dependencies (
       parent_alert_id INT,      # 根因告警
       child_alert_id INT,       # 衍生告警
       confidence FLOAT,         # 置信度 0-1
       detected_at TIMESTAMP
   );
   ```

3. **AI 根因推理**
   - Prompt 增强：分析多个告警的因果关系
   - 返回根因排序列表

4. **智能抑制**
   - 只通知根因告警
   - 衍生告警折叠显示："+ 99 个相关告警"

### 📊 预期价值

- ✅ 减少 90% 的噪音通知
- ✅ 快速定位根本问题
- ✅ 改善 oncall 体验

### 📋 开发清单

- [ ] 设计告警关系数据模型
- [ ] 实现时间窗口内的告警聚合
- [ ] AI Prompt 优化：因果关系推理
- [ ] 前端展示告警关系图
- [ ] 飞书通知格式调整

**预估工时**：3-5 天

```python
   → 正常，自动静默

✅ "每周一上午 流量激增"
   → 正常，只记录不通知

✅ "服务首次部署后的健康检查失败"
   → 正常，延迟5分钟通知
```

## 5. 成本优化 - AI 调用智能化

### 💡 核心创意

**问题**：每个告警都调用 AI，成本高、速度慢

### 🎯 解决方案

**分级分析策略**：
```python
告警来了 → 快速规则分析（0成本，<10ms）
  ├─ 低风险 → 只用规则（不调AI）✅ 省钱
  ├─ 中风险 → 批量分析（5个一起问AI，省90%）
  └─ 高风险 → 立即AI分析 ✅ 保证速度

窗口外重复：
  ├─ 告警内容相似度 > 95% → 不重新分析
  └─ 告警内容变化大 → 重新分析

缓存策略：
  相同 alert_hash + 相似的数值 → 复用AI分析
```

### 🛠️ 实现思路

1. **快速预分类**
   ```python
   def quick_classify(alert_data):
       # 规则引擎，0成本
       if 'Level' in alert_data:
           if alert_data['Level'] in ['critical', 'error']:
               return 'high'
           elif alert_data['Level'] == 'warn':
               return 'medium'
       return 'low'
   ```

2. **批量分析**
   ```python
   # 累积 5 个中风险告警，一次性发给 AI
   batch_queue = []
   if len(batch_queue) >= 5 or time_elapsed > 30s:
       ai_analyze_batch(batch_queue)
   ```

3. **相似度检测**
   ```python
   # 窗口外重复，检查内容相似度
   similarity = calculate_similarity(new_alert, old_alert)
   if similarity > 0.95:
       reuse_analysis(old_alert)
   else:
       ai_analyze(new_alert)
   ```

4. **分析缓存**
   ```python
   # 缓存 key: alert_hash + 数值范围
   cache_key = f"{alert_hash}:{round(value/10)*10}"
   ```

### 📊 预期价值

- ✅ AI 成本降低 80%
- ✅ 响应速度提升 3 倍
- ✅ 低风险告警秒级处理

### 📋 开发清单

- [ ] 规则引擎快速分类
- [ ] 实现批量分析队列
- [ ] 相似度计算算法
- [ ] AI 分析结果缓存
- [ ] 成本统计和监控

**预估工时**：1-2 天

## 7. 降级通知优化 - 分级通知策略

### 💡 核心创意

**问题**：AI 降级时的 24 小时限流太粗暴

### 🎯 解决方案

**渐进式通知**：
```python
AI降级场景分级：

第1次失败:
  → 静默降级，只记录日志

连续失败 3次 (15分钟内):
  → 发送内部通知（只给运维组）

连续失败 10次 (1小时内):
  → 升级通知 + 自动切换到备用AI服务

持续故障 24小时:
  → 暂时禁用AI，全部用规则分析
  → 每6小时尝试恢复一次
```

### 🛠️ 实现思路

1. **故障计数器**
   ```python
   # Redis 或内存中维护计数
   failure_count = redis.incr('ai_failure_count', expire=3600)
   ```

2. **分级通知**
   ```python
   if failure_count == 3:
       notify_internal()
   elif failure_count == 10:
       notify_all() + switch_to_backup()
   elif failure_count > 100:
       disable_ai() + schedule_retry()
   ```

3. **自动恢复**
   - 定期探测 AI 服务健康
   - 恢复后自动启用

4. **备用服务**
   - 配置多个 AI 服务地址
   - 主服务故障时自动切换

### 📊 预期价值

- ✅ 避免通知轰炸
- ✅ 智能自愈
- ✅ 高可用保障

### 🛠️ 实现思路

1. **指标计算**
   ```sql
   -- 告警总数趋势
   SELECT DATE(timestamp), COUNT(*)
   FROM webhook_events
   GROUP BY DATE(timestamp);

   SELECT AVG(resolved_at - timestamp)
   FROM webhook_events
   WHERE resolved_at IS NOT NULL;
   ```

   ```python
   score = (
       告警数量得分 * 0.3 +
       处理速度得分 * 0.3 +
       AI准确率得分 * 0.2 +
   ```

### 🎯 解决方案

**SLO 影响计算**：
```yaml
定义 SLO:
  服务: elys-backend
  SLI: 可用性 > 99.9%
  错误预算: 43分钟/月

告警自动计算影响：
  "本次故障持续 15分钟，消耗错误预算 34.8%"
  "本月剩余错误预算: 28分钟"

⚠️ 预警：错误预算剩余 < 20%
  → 通知团队谨慎发布
```

### 🛠️ 实现思路

1. **SLO 配置**
   ```yaml
   # config/slo.yaml
   services:
     elys-backend:
       sli_type: availability
       target: 99.9
       window: 30d
   ```

2. **错误预算计算**
   ```python
   total_budget = 30 * 24 * 60 * (1 - 0.999)  # 43分钟
   consumed = sum(故障时长)
   remaining = total_budget - consumed
   ```

3. **告警增强**
   - 展示对 SLO 的影响
   - 剩余预算预警

4. **报表**
   - 月度 SLO 达成情况

   ```python
   adapters = {
       'prometheus': PrometheusAdapter(),

   def handle_webhook(source, data):

       normalized = adapter.normalize(data)
       process(normalized)
   ```
